### 데이터(상태) 중심 설계

데이터(상태) 중심 설계는 객체의 상태, 즉 데이터를 중심으로 객체를 분할하고 설계하는 방법론입니다. 이 접근 방식에서는 객체의 상태가 구현의 핵심으로 간주됩니다. 하지만 문제는 구현은 불안정하며 변경 가능성이 크다는 점입니다. 따라서 상태를 중심으로 객체를 설계하면, 이 상태와 관련된 구현 세부사항이 객체의 인터페이스로 드러나게 되며, 이는 캡슐화 원칙을 훼손합니다.

캡슐화란 객체 내부의 구현 세부사항을 외부로부터 감추고, 외부에는 안정적인 인터페이스만을 제공하는 원칙을 말합니다. 그러나 데이터 중심 설계에서는 상태의 변경이 객체의 인터페이스 변경을 초래하게 되며, 이 인터페이스에 의존하는 다른 객체들도 변경의 영향을 받게 됩니다. 결과적으로 시스템 전체에 걸쳐 변경의 파급효과가 확대되기 쉽습니다. 이런 이유로 데이터 중심 설계는 변경에 취약하며, 시스템의 유지보수성을 떨어뜨릴 수 있습니다.

### 책임 중심 설계

책임 중심 설계는 객체의 책임을 중심으로 설계하는 접근 방식입니다. 여기서 책임이란 객체가 수행해야 할 행동이나 기능을 의미하며, 이는 객체의 인터페이스로 표현됩니다. 객체는 책임을 수행하기 위해 필요한 상태를 캡슐화하고, 외부에는 책임을 드러내는 인터페이스만을 노출시킵니다. 이를 통해 객체 내부의 구현이 변경되더라도, 외부에 미치는 영향은 최소화할 수 있습니다.

책임 중심 설계의 장점은 상대적으로 안정적인 설계를 얻을 수 있다는 점입니다. 객체는 내부의 상태나 구현 세부사항을 외부에 노출하지 않고, 인터페이스를 통해 외부와의 상호작용을 관리합니다. 이로 인해, 객체의 책임에 초점을 맞춘 설계는 변경에 강한 구조를 만들어내며, 시스템의 유지보수성을 높이는 데 기여합니다.

## 설계 트레이드 오프

### 캡슐화

캡슐화는 객체지향 설계에서 가장 중요한 원칙 중 하나입니다. 객체의 내부 구현을 외부로부터 감추고, 불안정한 부분(구현)과 안정적인 부분(인터페이스)을 명확히 분리함으로써 변경의 영향을 최소화할 수 있습니다. 캡슐화는 외부에서 알 필요가 없는 부분을 감춤으로써 대상을 단순화하는 추상화의 일종이기도 합니다.

설계의 품질을 판단하는 기준 중 하나는 얼마나 잘 캡슐화가 이루어졌는지입니다. 변경 가능성이 있는 요소들은 모두 캡슐화되어야 하며, 이를 통해 변경의 영향을 통제할 수 있습니다. 객체지향 설계의 핵심은 바로 이러한 캡슐화를 통해 객체의 내부 구현이 외부에 미치는 영향을 최소화하는 데 있습니다.

### 응집도와 결합도

응집도와 결합도는 객체지향 설계에서 모듈의 품질을 평가하는 중요한 척도입니다.

- **응집도**: 응집도는 모듈 내의 요소들이 얼마나 밀접하게 연관되어 있는지를 나타내는 지표입니다. 높은 응집도를 가진 모듈은 하나의 명확한 목적을 위해 구성된 요소들이 서로 긴밀하게 협력합니다. 반대로 낮은 응집도를 가진 모듈은 서로 다른 목적을 가진 요소들이 혼재해 있으며, 이는 유지보수성을 떨어뜨릴 수 있습니다.
- **결합도**: 결합도는 하나의 모듈이 다른 모듈에 얼마나 강하게 의존하고 있는지를 나타냅니다. 낮은 결합도는 모듈 간의 독립성을 높여, 변경이 발생하더라도 영향을 최소화할 수 있게 합니다. 반면, 높은 결합도는 한 모듈의 변경이 다른 모듈에 영향을 미치게 되어, 시스템 전체의 유지보수를 어렵게 만듭니다.

좋은 설계란 높은 응집도와 낮은 결합도를 가진 모듈로 구성된 시스템을 의미합니다. 캡슐화가 잘 이루어지면 자연스럽게 응집도는 높아지고 결합도는 낮아지게 됩니다. 따라서 캡슐화는 응집도와 결합도에 큰 영향을 미치는 중요한 요소로 작용합니다.

## 데이터(상태) 중심 설계의 문제점

### 캡슐화 위반

데이터 중심 설계에서는 객체의 상태가 주요 설계 기준이 되므로, 객체의 상태를 외부에 드러내는 접근자(Getter)와 수정자(Setter)가 많이 사용됩니다. 이로 인해 객체의 내부 상태가 인터페이스에 노출되며, 이는 캡슐화의 원칙을 위반하게 됩니다. 이러한 접근 방식은 객체가 다양한 상황에서 사용될 수 있다는 가정 하에 진행되며, 이는 결과적으로 불필요하게 많은 인터페이스를 노출하게 만들어 캡슐화를 무너뜨립니다.

결과적으로, 내부 구현이 변경될 때 이 변경이 외부로 퍼져나가게 되며, 이는 시스템의 안정성을 크게 저해할 수 있습니다. 이러한 문제는 데이터 중심 설계에서 캡슐화가 제대로 이루어지지 않았음을 나타내는 명백한 증거입니다.

### 높은 결합도

데이터 중심 설계는 객체의 내부 구현이 외부로 드러나게 만들어, 클라이언트가 그 구현에 강하게 결합되게 만듭니다. 이렇게 되면, 객체의 내부 구현이 변경될 때, 이 객체를 사용하는 모든 클라이언트가 함께 변경되어야 합니다. 이는 시스템 전체에 걸쳐 변경의 파급 효과를 확대시킵니다.

또한, 제어 로직이 특정 객체에 집중되는 경향이 있기 때문에, 제어 객체와 다수의 데이터 객체가 강하게 결합됩니다. 이로 인해, 하나의 데이터 객체를 변경하더라도 제어 객체를 포함한 다수의 객체가 함께 변경되어야 하는 상황이 발생할 수 있습니다.

### 낮은 응집도

데이터 중심 설계는 모듈 내부에 변경의 이유가 서로 다른 코드들을 혼합하는 경향이 있습니다. 이로 인해 모듈의 응집도가 낮아지며, 하나의 변경이 아무 관련 없는 코드에까지 영향을 미치는 문제가 발생할 수 있습니다.

응집도가 낮은 모듈은 특정 요구사항이 변경될 때, 그 요구사항과 관련 없는 코드들도 함께 수정해야 하는 상황을 만들 수 있습니다. 이는 유지보수성을 떨어뜨리고, 시스템의 복잡성을 증가시킵니다.

### 데이터 중심 설계의 추가적인 문제점

데이터 중심 설계에서는 객체의 상태를 먼저 결정하고, 이후에 그 상태를 관리하기 위한 오퍼레이션을 정의하는 경향이 있습니다. 하지만 이 방식은 객체의 캡슐화를 무너뜨리고, 객체의 인터페이스를 복잡하게 만듭니다.

또한, 데이터 중심 설계는 객체의 내부에 초점을 맞추기 때문에, 객체가 다른 객체들과 협력하는 방식을 고려하지 않고 설계될 가능성이 큽니다. 이는 객체의 인터페이스가 이미 결정된 상태에서 다른 객체와의 협력 방법을 억지로 끼워 맞추게 만들며, 협력이 객체의 내부 구현 세부사항에 종속되게 합니다.

이로 인해 객체의 내부 구현이 변경되었을 때, 협력하는 모든 객체들이 영향을 받게 되는 문제를 초래합니다. 이러한 문제는 시스템의 안정성과 유지보수성을 크게 저해합니다.

## 자율적인 객체를 향해

### 캡슐화를 지켜라

데이터 중심 설계가 문제를 일으키는 근본적인 이유는 캡슐화의 원칙을 위반했기 때문입니다. 객체는 자신이 어떤 데이터를 가지고 있는지를 외부에 노출해서는 안 됩니다. 객체는 자신의 상태를 책임지고 관리해야 하며, 외부에서는 인터페이스에 정의된 메서드를 통해서만 상태에 접근할 수 있어야 합니다.

여기서 중요한 점은, 객체가 제공하는 메서드는 단순히 속성 값을 반환하거나 수정하는 것이 아닌, 객체가 책임져야 하는 의미 있는 작업을 수행하는 메서드여야 한다는 것입니다. 객체는 스스로의 상태를 책임지는 자율적인 존재로 설계되어야 하며, 외부에서 객체의 내부 상태에 직접 접근하는 방식은 지양해야 합니다.

### 스스로 자신의 데이터를 책임지는 객체

상태와 행동을 하나의 단위로 묶는 이유는 객체가 스스로 자신의 상태를 처리할 수 있도록 하기 위함입니다. 객체는 단순히 데이터를 저장하고 전달하는 역할에 그쳐서는 안 되며, 그 데이터를 처리하는 데 필요한 책임을 스스로 수행할 수 있어야 합니다.

따라서 객체를 설계할 때는 객체가 어떤 데이터를 포함해야 하는지와 더불어, 그 데이터에 대해 수행해야 할 오퍼레이션이 무엇인지를 명확히 정의해야 합니다. 이렇게 정의된 오퍼레이션 집합이 객체의 인터페이스를 형성하며, 이를 통해 객체는 자율적인 존재로서 시스템 내에서의 역할을 수행하게 됩니다.

이러한 접근 방식은 객체지향 설계의 기본 원칙을 따르며, 시스템의 유지보수성을 높이고 변경에 강한 구조를 만들어냅니다.
