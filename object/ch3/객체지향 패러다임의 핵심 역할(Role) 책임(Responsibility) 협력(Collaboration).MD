# 객체지향 패러다임의 핵심: 역할(Role), 책임(Responsibility), 협력(Collaboration)

## 협력(Collaboration)

### 정의 및 중요성

- **협력**은 객체들이 시스템의 기능을 구현하기 위해 서로 상호작용하는 과정입니다. 객체는 혼자서 모든 기능을 구현할 수 없으므로, 다른 객체와 협력하여 시스템 전체의 목표를 달성합니다.
- 객체지향 패러다임의 본질은 협력하는 객체들의 공동체를 구성하는 것에 있습니다. 이는 적절한 객체를 찾고, 그 객체들에게 적절한 책임을 할당하는 과정을 통해 이루어집니다. 협력은 시스템이 유연하고 변경에 강한 구조를 가지도록 돕습니다.

### 협력의 시작과 메시지 전송

- 협력은 하나의 객체가 다른 객체에게 도움을 요청하는 행위에서 시작됩니다. 이때 **메시지 전송**이 협력을 위해 객체 간에 사용할 수 있는 유일한 커뮤니케이션 수단이 됩니다.
- 메시지를 수신한 객체는 **메서드**를 실행하여 요청에 응답합니다. 여기서 중요한 점은, 메시지를 처리하는 방법을 수신한 객체가 스스로 선택한다는 것입니다. 외부의 객체는 단지 메시지만 전송할 수 있을 뿐, 그 메시지가 어떻게 처리될지는 전적으로 수신 객체의 자율적인 결정에 달려 있습니다.

### 객체의 자율성

- 객체는 **자율적인 존재**로, 자신의 일을 스스로 처리할 수 있어야 합니다. 이를 위해 객체는 필요한 정보와 그 정보에 기반한 행동을 내부에 갖추고 있어야 하며, 이는 **캡슐화**를 통해 구현됩니다.
- 캡슐화는 객체의 내부 구현을 외부로부터 숨겨, 객체의 자율성을 보장하는 동시에 변경의 파급효과를 줄이는 중요한 메커니즘입니다. 이렇게 함으로써 객체들은 자신의 상태와 행동을 독립적으로 관리할 수 있습니다.

### 협력의 문맥과 객체 설계

- 협력은 객체 설계를 위한 **문맥(context)**을 제공합니다. 애플리케이션 내에서 어떤 객체가 필요한 이유는 오직 그 객체가 특정 협력에 참여하기 때문입니다. 객체는 협력에 필요한 행동을 제공할 수 있어야 하며, 그 행동이 바로 객체가 존재하는 이유가 됩니다.
- 객체의 행동은 협력에 의해 결정되며, 이 협력이 바뀌면 객체가 제공해야 하는 행동도 바뀌어야 합니다. 이는 객체가 가진 상태 역시 협력의 필요에 따라 결정되어야 한다는 것을 의미합니다. 객체의 상태는 객체가 행동을 수행하는 데 필요한 정보에 의해 정의됩니다.

### 협력과 애플리케이션의 제어 흐름

- 객체지향 애플리케이션에서 제어 흐름은 특정 하나의 객체에 의해 통제되지 않고, 다양한 객체들 사이에 균형 있게 분배되는 것이 일반적입니다. 이는 시스템을 더 유연하게 만들고, 변경에 쉽게 대응할 수 있도록 합니다.
- 협력하는 객체들은 서로 메시지를 주고받으며 상호작용하고, 이를 통해 애플리케이션의 기능이 구현됩니다. 이처럼 객체들 간의 협력은 시스템의 기능을 구현하는 데 필수적인 요소입니다.

## 책임(Responsibility)

### 정의 및 역할

- **책임**은 객체가 협력에 참여하기 위해 수행해야 하는 로직이나 행동을 의미합니다. 객체는 협력에서 맡은 역할을 충실히 수행하기 위해 적절한 책임을 지닙니다.
- 크레이그 라만(Craig Larman)은 책임을 크게 두 가지로 나눕니다: **하는 것(Doing)**과 **아는 것(Knowing)**. 이는 객체가 무엇을 알고, 무엇을 할 수 있는지를 기준으로 책임을 정의합니다.

### 하는 것(Doing)

- 객체가 **스스로 수행할 수 있는 행동**을 포함합니다. 예를 들어, 객체를 생성하거나 계산을 수행하는 것, 다른 객체의 행동을 시작시키는 것, 또는 다른 객체의 활동을 제어하고 조절하는 것 등이 이에 해당합니다.

### 아는 것(Knowing)

- 객체가 **알고 있어야 하는 정보**를 포함합니다. 이는 사적인 정보나 관련된 다른 객체에 대한 지식, 또는 자신이 유도하거나 계산할 수 있는 정보 등을 의미합니다.

### 책임 할당

- 객체의 책임은 협력이라는 문맥 속에서 결정되며, 협력에 필요한 행동을 수행할 적절한 객체를 찾아 그 책임을 할당해야 합니다. 이때 중요한 원칙 중 하나는 **정보 전문가(Information Expert)** 패턴입니다. 이는 특정 정보를 가장 잘 알고 있는 객체에게 그 책임을 할당하는 것을 의미합니다.
- 시스템이 사용자에게 제공해야 하는 기능을 하나의 책임으로 보고, 이를 더 작은 책임으로 분할하여 적절한 객체에 할당하는 과정이 반복됩니다. 이 과정에서 객체는 협력에 필요한 메시지를 찾고, 메시지에 적절한 객체를 선택합니다. 이렇게 할당된 메시지는 객체의 책임을 결정하게 됩니다.

### 책임 주도 설계(RDD)

- **책임 주도 설계(Responsibility-Driven Design, RDD)** 는 협력을 설계하기 위해 책임에 초점을 맞추는 설계 방식입니다. 책임 주도 설계의 핵심은 시스템의 기능을 작은 책임들로 나누고, 이를 적절한 객체에 할당하는 것입니다.
- 이 과정에서 협력은 객체의 행동과 상태를 결정하는 문맥을 제공하며, 이를 통해 시스템의 기능을 구현할 수 있습니다.

### 책임 할당의 구체적 과정

1. **시스템 책임 파악**: 시스템이 사용자에게 제공해야 하는 기능, 즉 시스템 책임을 파악합니다.
2. **책임 분할**: 시스템 책임을 더 작은 책임으로 분할합니다.
3. **책임 할당**: 분할된 책임을 수행할 수 있는 적절한 객체나 역할을 찾아 책임을 할당합니다.
4. **협력 설계**: 객체가 책임을 수행하는 도중 다른 객체의 도움이 필요한 경우, 적절한 객체를 찾아 협력하게 합니다. 이 과정을 통해 객체 간 협력이 형성됩니다.

### 책임과 객체의 인터페이스

- 객체에 할당된 책임은 그 객체의 외부 인터페이스와 내부 속성을 결정합니다. 책임이 어떻게 할당되는지에 따라 객체의 퍼블릭 인터페이스가 정의되며, 이는 객체가 외부와 상호작용하는 방식을 결정합니다.
- 책임은 메시지보다 더 추상적이고 개념적인 것이며, 처음에는 단순해 보였던 책임도 나중에는 여러 메시지로 분할되거나, 여러 객체가 협력해야만 수행할 수 있는 커다란 책임으로 성장할 수 있습니다.

### 책임과 협력의 상호작용

- 협력은 객체에게 적절한 책임을 이끌어내고, 이 책임이 객체의 행동과 상태를 결정합니다. 객체는 협력 속에서 자신의 책임을 수행하며, 필요한 경우 다른 객체에게 도움을 요청해 협력을 강화합니다.
- 적절한 책임 할당을 통해 자율적인 객체를 만들 수 있으며, 이는 전체 시스템의 유연성과 확장성을 높이는 데 기여합니다.
